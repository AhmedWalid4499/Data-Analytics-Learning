1 · The relational model

Tables, rows, columns & keys

A relational database stores data in tables (also called relations). Each row is one record (a tuple); each column is one attribute. The schema is the blueprint — which tables exist, their columns, data types and rules.

Primary key (PK) — a column (or set of columns) that uniquely identifies each row. order_id in an Orders table. No duplicates, never empty.
Foreign key (FK) — a column that points to the primary key of another table. This is what creates a relationship. customer_id in Orders points to customer_id in Customers.
Composite key — a primary key made of two or more columns together, used when no single column is unique on its own.
Surrogate key — an artificial key (often an auto-number) used as the PK instead of "natural" data, so the key never has to change.

How tables relate

Relationships come in three shapes. The trick with many-to-many is that relational databases can't store it directly — you resolve it with a junction (bridge) table holding two foreign keys.

RelationshipMeaningDPM example
One-to-one (1:1)A row here matches at most one row thereAn Order has one delivery Sign-off record
One-to-many (1:N)One row here matches many rows thereOne Customer has many Orders
Many-to-many (M:N)Many match many — needs a junction tableEngineers ↔ Sites (an engineer covers many sites; a site is covered by many engineers)

Normalization, in one breath

Normalization means organising tables so each fact is stored once, to avoid update anomalies and redundancy.

1NF — every cell holds a single, atomic value; no repeating groups or comma-lists in a column.
2NF — 1NF, plus every non-key column depends on the whole primary key (matters only for composite keys).
3NF — 2NF, plus non-key columns depend on the key and nothing but the key (no column derived from another non-key column).
i
Transactional systems aim for 3NF to keep writes clean. Analytics systems often denormalize on purpose — trading some redundancy for fewer joins and faster reads. That tension is exactly what star vs snowflake is about (section 4).
2 · Under the hood

How a database actually stores data: pages

You query rows, but the database doesn't read rows one at a time. It reads and writes data in fixed-size blocks called pages (8 KB in SQL Server and PostgreSQL; "blocks" in Oracle). The page is the smallest unit of I/O — grab one row and the engine actually loads the whole page it lives on.

One row request pulls a whole page. An index finds which page holds your row so the engine doesn't scan every page; the buffer pool keeps hot pages in RAM so repeat reads skip the disk entirely.

Heap vs clustered — a table with no clustered index is a "heap" (rows in no order). A clustered index physically sorts rows by the key, so range scans read fewer pages.
Index — usually a B-tree. Like the index at the back of a book: it tells the engine the exact page, turning a full scan into a quick lookup. Great for reads, a small cost on writes.
Buffer pool / cache — pages already read are held in memory. This is why the second run of a query is often much faster than the first.
ACID — the guarantee transactional systems give: Atomicity, Consistency, Isolation, Durability. A transaction either fully happens or not at all, even if the power fails mid-write.
i
As an analyst you mostly read from these systems. Knowing about pages and indexes explains the practical stuff: why a missing index makes a query crawl, why SELECT * on a billion-row table is painful, and why analytics often copies data into a warehouse shaped for reading rather than writing.
3 · ER diagrams

ERD — mapping entities & relationships

An Entity-Relationship Diagram is the picture of your data model. Entities are the things you store (Customer, Order, Site). Attributes are their properties. Relationships are the lines between them, annotated with cardinality — how many of one relate to how many of another.

Read it as: one Customer places many Orders; each Order is delivered at one Site. The single tick means "one", the fork (crow's foot) means "many". The FK columns in Order are what physically wire it together.

Crow's-foot cardinality, decoded

Symbol at the end of a lineMeans
│ (single bar)Exactly one
○ (circle)Zero (optional)
< (fork / crow's foot)Many
○< Zero or many
│< One or many
││ One and only one

Try it. The full DPM dataset also tracks a DPM (project manager) and a Field Engineer per order. How would you add them to the ERD above, and what cardinality fits "a DPM manages many orders"?

Reveal a model answer

Add two more entities — DPM and FieldEngineer — each with its own PK. Give Order two more foreign keys: dpm_id and fe_id. The relationships are one-to-many in both cases: one DPM manages many orders (DPM │——< Order), one field engineer works many orders. If, in reality, an engineer can cover many sites and a site needs many engineers, that Engineer↔Site link is many-to-many and would need a junction table (e.g. SiteCoverage holding site_id + fe_id).

4 · Enhanced ER diagrams

EERD — when plain ERD isn't enough

An Enhanced (or Extended) ERD adds object-oriented ideas on top of the basic ERD so you can model richer real-world structure. The big one is generalization / specialization — a supertype that shares attributes with its subtypes.

Generalization in action. DPM and Field Engineer are both Employees — they inherit emp_id, name, email — but each adds its own attributes. The circle marks the specialization; d = disjoint.

The EERD concepts worth knowing

Generalization / Specialization (ISA) — a supertype and its subtypes. Subtypes inherit the supertype's attributes and relationships.
Disjoint (d) vs Overlapping (o) — disjoint: an instance belongs to one subtype only. Overlapping: it can belong to several (e.g. someone who is both a DPM and an Engineer).
Total vs Partial participation — total (shown as a double line): every supertype instance must be in some subtype. Partial: it may belong to none.
Aggregation — treating a whole relationship as a single higher-level entity so another relationship can connect to it.
Union / Category — a subtype whose members come from a union of several different supertypes.
5 · Data-warehouse modeling

Star vs Snowflake schema

For analytics you reshape data into fact and dimension tables. A fact table holds the events you measure — one row per order — with numeric measures and foreign keys. Dimension tables hold the descriptive context: who, what, where, when. How you arrange the dimensions gives you a star or a snowflake.

★ Star schema

Each dimension is a single table, one join from the fact. Simple, fast, few joins.

❄ Snowflake schema

Dimensions are normalized into sub-tables (Site → City → Country → Region). Less redundancy, more joins.

Which should you use?

Star ★Snowflake ❄
DimensionsDenormalized (one table each)Normalized into sub-tables
Joins to answer a queryFewer (one hop)More (multi-hop)
Query speedFasterOften slower
StorageSlightly more (repeated text)Less
Simplicity for report buildersHighLower
Best forPower BI & most BI workVery large dims or strict hierarchy control
i
Rule of thumb: in Power BI, prefer a star schema. The engine (VertiPaq) is built for it, DAX is simpler, and relationships stay easy to reason about. Snowflake only when a dimension is genuinely huge or a shared hierarchy must be enforced once. This connects straight to the Power BI track, where a clean star makes your measures behave.
6 · OLAP

OLAP — analysing data by dimensions

The systems that run the business (writing one order at a time) are OLTP. The systems you analyse with — slicing totals by region, by quarter, by service — are OLAP. Power BI's model is an OLAP-style layer sitting on top of your star schema.

OLTP (transactional)OLAP (analytical)
PurposeRun operationsAnalyse & decide
Typical queryInsert/update one orderSum revenue by region by quarter
Data shapeNormalized (3NF)Star / cube, denormalized
Reads vs writesMany small writesFew large reads
Example hereThe GOLD order portalYour Power BI KPI dashboard

The OLAP cube

Think of measures sitting inside a cube whose edges are dimensions. Each cell is a pre-aggregated number — e.g. revenue for EMEA / SD-WAN / Q1.

One cell = one answer. Slide along an axis and you're changing the question — by service, by quarter, by region.

The five OLAP operations

Roll-up — aggregate up a hierarchy. Site totals → Country → Region. "Show revenue by region instead of by site."
Drill-down — the reverse. Region → Country → Site. "EMEA looks high — break it down by site."
Slice — fix one dimension to a single value, taking a flat slab of the cube. "Just Q1, all regions, all services."
Dice — pick a sub-cube across several dimensions. "Q1–Q2, SD-WAN + LAN only, EMEA + APAC."
Pivot (rotate) — reorient the axes. Swap services-as-rows for regions-as-rows to read the same data a different way.

The questions OLAP is built to answer

If a business question sounds like "measure by dimension by dimension", it's an OLAP question. A few from the DPM world:

QuestionOperation
Total revenue by region this yearRoll-up
That region is high — show it by siteDrill-down
On-time delivery % for Q1 onlySlice
FTE days for SD-WAN + LAN across EMEA & APAC, H1Dice
Flip the table: services down the side, quarters across the topPivot
