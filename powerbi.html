▸Orientation Desktop, Service & the build order Basics

Desktop vs Service

Power BI Desktop (free, on your PC) is where you build. Power BI Service (powerbi.com) is where you publish so others can view and where scheduled refresh lives.

The three things you'll make

A semantic model (the data + relationships + measures), reports (interactive multi-visual pages), and dashboards (pinned highlights, Service-only).

The build order — every report follows this

1 · GET DATA Connect to the Excel/CSV/web/SQL source
2 · TRANSFORM Clean & shape it in Power Query (same engine as Excel)
3 · MODEL Create relationships between tables, set data types
4 · DAX Write measures & calculated columns
5 · VISUALIZE Choose visuals, add slicers, arrange the page
6 · PUBLISH Push to the Service, schedule refresh, share

▸Data model Relationships & the star schema Model

In the Model view you connect tables so a filter on one flows to another. The clean pattern is a star schema: one central fact table of events surrounded by dimension tables that describe them.

Fact table

The events you measure — here, the Projects rows (each order, with revenue, FTE, dates). Lots of rows, mostly numbers and keys.

Dimension tables

The descriptors you slice by — a Date table, a Customer table, a Region table. Fewer rows, used for filtering and grouping.

★
Almost always add a dedicated Date table and relate it to your date column. Time-intelligence DAX (YTD, prior month) depends on it.
▸Chart chooser Pick the visual by the question Visualize

A chart's job is to make one comparison obvious. Pick by the question you're answering — not by what looks fanciest. Here's when each earns its place, and when it misleads.

Bar / Column

Use for: comparing a value across categories (revenue by customer, orders by region). The default, and usually the right answer. Bars (horizontal) suit long category names.

Line

Use for: a trend over time (orders per month, revenue by quarter). Continuous time on the X axis. Don't use a line for unordered categories.

Stacked column

Use for: a total and its parts (revenue per region split by service). Great for the total; hard to compare the middle segments — use 100% stacked if proportion is the point.

Pie / Donut

Use sparingly: one part-to-whole split with 2–4 slices (Delivered vs not). Avoid with many slices or for comparing similar sizes — a bar chart is clearer almost every time.

Scatter

Use for: relationship between two numbers (FTE days vs revenue) and spotting outliers. Add a third measure as bubble size for "scatter+".

Treemap

Use for: part-to-whole when you have many categories and only care about the big ones (revenue share by customer). Hard to read precise values — pair with a tooltip.

Card / KPI

Use for: a single headline number (total revenue, delivered %). The KPI variant adds a target and trend. Put these top-left — eyes land there first.

Matrix / Table

Use for: precise numbers people will read or export, and drill-down by rows+columns (Region × Service). When the exact figure matters more than the shape, use a matrix.

Gauge

Use rarely: one value against a target (delivered % vs goal). Eats space for one number — a KPI card usually does the job with less ink.

✓
Interaction is free: click a bar and every other visual on the page filters to it. Add Slicers (Region, Status, Date) so viewers explore without you. That responsiveness is the whole point of Power BI over a static chart.

▸DAX Measures that react to filters DAX

DAX (Data Analysis Expressions) is the formula language. The single most important distinction:

Calculated column

Computed per row, stored in the table, fixed at refresh. Use for row-level attributes you'll group or filter by.

Days Late = IF( ISBLANK(Projects[Actual Go-Live]), BLANK(), Projects[Actual Go-Live] - Projects[Target Go-Live] )

Measure

Computed on the fly for whatever's on screen, reacting to every slicer and click. Use for all aggregations and KPIs.

Total Revenue = SUM( Projects[Revenue (EUR)] )
!
Beginner trap: don't build measures as calculated columns. If you find yourself wanting a total or average that should change with filters, it's a measure.

The measures you'll write first

Order Count = COUNTROWS( Projects ) Delivered Orders = CALCULATE( [Order Count], Projects[Status] = "Delivered" ) Delivered % = DIVIDE( [Delivered Orders], [Order Count] ) Avg FTE Days = AVERAGE( Projects[FTE Days] ) Distinct Customers= DISTINCTCOUNT( Projects[Customer] ) EMEA Revenue = CALCULATE( [Total Revenue], Projects[Region] = "EMEA" )

CALCULATE is the heart of DAX — it changes the filter context an expression runs in. DIVIDE is safe division (returns blank, not an error, on divide-by-zero).

Time intelligence (needs a Date table)

Revenue YTD = TOTALYTD( [Total Revenue], 'Date'[Date] ) Revenue Last Month= CALCULATE( [Total Revenue], PREVIOUSMONTH( 'Date'[Date] ) )

Try it

Write two measures: the number of On Hold orders, and the average revenue per delivered order.

Show answer
On Hold Orders = CALCULATE( [Order Count], Projects[Status] = "On Hold" ) Avg Delivered Rev = DIVIDE( CALCULATE( [Total Revenue], Projects[Status]="Delivered" ), [Delivered Orders] )
Good habits

What separates a clean report

Measures, not columns — if a total or average should react to slicers, it's a measure. Reserve calculated columns for row-level attributes you group or filter by.
Star schema first — one fact table surrounded by dimensions. The engine and DAX are built for it; a single flat table fights you later.
Always add a Date table — and mark it as one. Every time-intelligence measure (YTD, prior month, YoY) depends on it.
DIVIDE, not / — DIVIDE(a, b, 0) returns a blank or your fallback instead of a divide-by-zero error.
One visual, one question — pick the chart by the comparison you're making, then add slicers and let interactions do the exploring.
Name & format measures — explicit, well-named, pre-formatted measures keep a model readable and reusable.
