Clauses by level

Build your SQL vocabulary

SQL reads and summarises data in relational tables — the model from the Foundations page. This track is split into three levels — tap any card to switch. Start at Level 1 and climb. Each clause has a worked example on the orders table and a task to try yourself — paste your answer into the runner below to check it.

▸SELECT / FROM Choose columns Basic

Pick the columns you want from a table. * means "all columns"; otherwise list names separated by commas.

SELECT customer, region, revenue FROM orders;

Try it

Return just the site_code and city for every order.

Show answer
SELECT site_code, city FROM orders;

Column names go after SELECT; the table after FROM.

▸WHERE Filter rows Basic

Keep only the rows that match a condition. Text values go in 'single quotes'.

SELECT customer, revenue FROM orders WHERE region = 'EMEA' AND status = 'Delivered';
i
Operators: = equals, <> not equal, > < >= <=, and AND / OR to combine conditions.

Try it

List the customer and revenue of every order in the APAC region.

Show answer
SELECT customer, revenue FROM orders WHERE region = 'APAC';
▸ORDER BY / LIMIT Sort and cap Basic

Sort by a column (DESC for high-to-low, ASC for low-to-high) and keep just the top few with LIMIT.

SELECT customer, site_code, revenue FROM orders ORDER BY revenue DESC LIMIT 5;

Try it

Show the 3 smallest orders by revenue (customer and revenue).

Show answer
SELECT customer, revenue FROM orders ORDER BY revenue ASC LIMIT 3;

ASC is the default sort, but spelling it out makes intent clear.

▸DISTINCT Unique values Basic

Remove duplicates so each value appears once — handy for "what values exist in this column?".

SELECT DISTINCT country FROM orders ORDER BY country;

Try it

List each service_type exactly once.

Show answer
SELECT DISTINCT service_type FROM orders;
▸Aggregates + GROUP BY Summarise Intermediate

COUNT, SUM, AVG, MIN and MAX collapse many rows into one. GROUP BY does it per category, and AS renames the output column.

SELECT region, COUNT(*) AS orders, ROUND(SUM(revenue)) AS total_rev, ROUND(AVG(fte_days),1) AS avg_fte FROM orders GROUP BY region ORDER BY total_rev DESC;

Try it

For each service_type, show the number of orders and total revenue, busiest first.

Show answer
SELECT service_type, COUNT(*) AS orders, ROUND(SUM(revenue)) AS total_rev FROM orders GROUP BY service_type ORDER BY orders DESC;
▸HAVING Filter groups Intermediate

WHERE filters rows before grouping; HAVING filters the groups after.

SELECT dpm, COUNT(*) AS orders FROM orders GROUP BY dpm HAVING COUNT(*) >= 10 ORDER BY orders DESC;
i
Order of execution: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. That's why a SELECT alias can't be used in WHERE — it doesn't exist yet.

Try it

Which customers have more than 8 orders? Show the customer and the count.

Show answer
SELECT customer, COUNT(*) AS orders FROM orders GROUP BY customer HAVING COUNT(*) > 8 ORDER BY orders DESC;
▸JOIN Combine tables Intermediate

Match rows from two tables on a shared key. Here we compare each region's actual revenue to its target from the targets table.

SELECT o.region, ROUND(SUM(o.revenue)) AS actual, t.revenue_target, ROUND(100.0 * SUM(o.revenue) / t.revenue_target) AS pct_of_target FROM orders o JOIN targets t ON o.region = t.region GROUP BY o.region ORDER BY actual DESC;

Try it

Show each region with its actual total revenue and its revenue_target side by side.

Show answer
SELECT o.region, ROUND(SUM(o.revenue)) AS actual, t.revenue_target FROM orders o JOIN targets t ON o.region = t.region GROUP BY o.region;

Table aliases (o, t) keep the column references short and unambiguous.

▸CASE If-then logic in a column Advanced

Bucket rows into categories on the fly, then group by the result.

SELECT CASE WHEN revenue >= 30000 THEN 'large' WHEN revenue >= 15000 THEN 'mid' ELSE 'small' END AS band, COUNT(*) AS orders FROM orders GROUP BY band ORDER BY orders DESC;

Try it

Label each order 'High' when priority is 'High', otherwise 'Other', and count each label.

Show answer
SELECT CASE WHEN priority = 'High' THEN 'High' ELSE 'Other' END AS bucket, COUNT(*) AS n FROM orders GROUP BY bucket;
▸Subquery A query inside a query Advanced

Use the result of one query inside another. Here: orders above the overall average revenue.

SELECT customer, site_code, revenue FROM orders WHERE revenue > (SELECT AVG(revenue) FROM orders) ORDER BY revenue DESC;

Try it

List orders whose revenue is above the average revenue of EMEA orders.

Show answer
SELECT customer, revenue FROM orders WHERE revenue > (SELECT AVG(revenue) FROM orders WHERE region = 'EMEA') ORDER BY revenue DESC;
▸CTE (WITH) Name a step for readability Advanced

A common table expression is a named temporary result you can build on — cleaner than nesting subqueries.

WITH region_rev AS ( SELECT region, SUM(revenue) AS total FROM orders GROUP BY region ) SELECT region, total FROM region_rev ORDER BY total DESC;

Try it

Using a CTE of orders-per-DPM, return only the DPMs with 10 or more orders.

Show answer
WITH c AS ( SELECT dpm, COUNT(*) AS n FROM orders GROUP BY dpm ) SELECT * FROM c WHERE n >= 10 ORDER BY n DESC;

Because the CTE already computed n, you can filter on it with a plain WHERE.

▸Window functions Rank without collapsing rows Advanced

Aggregate across rows while keeping every row. RANK() OVER (...) ranks each order by revenue within its region.

SELECT region, customer, revenue, RANK() OVER (PARTITION BY region ORDER BY revenue DESC) AS rev_rank FROM orders ORDER BY region, rev_rank LIMIT 15;

Try it

Rank all orders by revenue (highest = 1), showing customer, revenue and the rank.

Show answer
SELECT customer, revenue, RANK() OVER (ORDER BY revenue DESC) AS rnk FROM orders ORDER BY rnk LIMIT 15;

Drop PARTITION BY to rank across the whole table instead of within each group.

Create your own

Databases, tables & inserting data

So far you've queried tables that already exist. This is the other half of SQL: building them — creating a database, defining tables with the right data types and rules, and loading rows in. The statements that define structure (CREATE, ALTER, DROP) are called DDL; the ones that change data (INSERT, UPDATE, DELETE) are DML.

i
The runner below is SQLite, which treats the whole session as a single database — so there's no CREATE DATABASE step there. Start at CREATE TABLE and the sample script at the end runs as-is. (Refresh the page to reset.)

1 · Databases

On a server engine you first create a database to hold your tables, then switch into it.

-- On a server engine (MySQL / SQL Server) CREATE DATABASE bookshop; USE bookshop; -- PostgreSQL uses: \c bookshop -- ...create tables and load data here... DROP DATABASE bookshop; -- deletes the whole database — careful!
i
Dialect note: SQLite has no CREATE DATABASE — each .db file is a database, so you just open the file and start with CREATE TABLE. To use a second file from the same session: ATTACH DATABASE 'archive.db' AS archive;

2 · Column data types

Every column has a type that says what it can hold. The exact names vary slightly by engine; these are the common ones.

TypeHoldsExample
INTEGER / INTWhole numbers — ids, counts42
REAL / FLOAT / DOUBLEApproximate decimals3.14
DECIMAL(p,s) / NUMERICExact decimals — money19.99
TEXTVariable-length text (any length)'hello'
VARCHAR(n)Variable text up to n characters'Acme Ltd'
CHAR(n)Fixed-length text, padded to n'US' (CHAR(2))
DATE / DATETIME / TIMESTAMPDates and times'2025-01-31'
BOOLEANTrue / false (often stored as 1 / 0)1
BLOBRaw binary (images, files)—
i
CHAR vs VARCHAR: CHAR(n) always stores exactly n characters (short values are space-padded) — good for fixed codes like a 2-letter country. VARCHAR(n) stores only what you put in, up to n. SQLite is loose here — it uses simple type affinity (TEXT, INTEGER, REAL, NUMERIC, BLOB) and won't enforce the length, but writing VARCHAR(200) still documents intent and is portable.

3 · Constraints — the rules on a table

Constraints keep the data valid. They're declared right inside CREATE TABLE.

ConstraintWhat it enforces
PRIMARY KEYUniquely identifies each row (unique and not null). One per table.
AUTOINCREMENTAuto-generates the next id. SQLite: INTEGER PRIMARY KEY AUTOINCREMENT; MySQL AUTO_INCREMENT; SQL Server IDENTITY(1,1); PostgreSQL SERIAL.
NOT NULLA value is required — the column can't be left empty.
UNIQUENo two rows may share the same value.
DEFAULT vUses v when no value is supplied on insert.
CHECK (cond)Rejects any row where the condition is false (e.g. price >= 0).
FOREIGN KEY (c) REFERENCES t(c)The value must match an existing key in another table (referential integrity).
PRIMARY KEY (a, b)A composite key — uniqueness across several columns together.

4 · CREATE TABLE

Put it together: each column is name TYPE [constraints], separated by commas, with any table-level rules (like a foreign key) at the end.

-- Parent table: one row per author CREATE TABLE authors ( author_id INTEGER PRIMARY KEY, -- unique id for each author name TEXT NOT NULL, -- required country CHAR(2) -- 2-letter code, e.g. 'US' ); -- Child table: each book belongs to one author CREATE TABLE books ( book_id INTEGER PRIMARY KEY AUTOINCREMENT, -- id fills itself title VARCHAR(200) NOT NULL, author_id INTEGER, genre TEXT DEFAULT 'General', -- used when none given price REAL CHECK (price >= 0), -- can't be negative in_stock INTEGER DEFAULT 0, published DATE, FOREIGN KEY (author_id) REFERENCES authors(author_id) );
!
SQLite only enforces foreign keys after you run PRAGMA foreign_keys = ON; once per connection (it's off by default for backwards-compatibility). Most server engines enforce them always.

5 · INSERT — several ways

a. Name the columns (the safest, clearest way — order doesn't matter):

INSERT INTO authors (author_id, name, country) VALUES (4, 'Toni Morrison', 'US');

b. All columns, in table order (no column list — you must supply every column):

INSERT INTO authors VALUES (5, 'Gabriel Garcia Marquez', 'CO');

c. Several rows in one statement:

INSERT INTO authors (author_id, name) VALUES (6, 'Italo Calvino'), (7, 'Octavia Butler');

d. Let defaults and AUTOINCREMENT do the work — list only what you have; book_id auto-fills and genre/in_stock take their defaults:

INSERT INTO books (title, author_id, price) VALUES ('Beloved', 4, 11.99);

e. Insert the result of a query (INSERT … SELECT) — copy rows from one table into another:

INSERT INTO cheap_books (title, price) SELECT title, price FROM books WHERE price < 11;
i
Inserts add rows. To change existing rows use UPDATE … SET … WHERE …; to remove them use DELETE FROM … WHERE …. Always include a WHERE, or you'll change/clear every row.

6 · A complete sample script

This builds two related tables, loads them with the different insert styles, copies a filtered set into a third table, and finishes with a join to prove it worked. Paste the whole thing into the runner below and run it.

-- Enforce foreign keys (SQLite has them off by default) PRAGMA foreign_keys = ON; -- 1) Define the tables CREATE TABLE authors ( author_id INTEGER PRIMARY KEY, name TEXT NOT NULL, country CHAR(2) ); CREATE TABLE books ( book_id INTEGER PRIMARY KEY AUTOINCREMENT, title VARCHAR(200) NOT NULL, author_id INTEGER, genre TEXT DEFAULT 'General', price REAL CHECK (price >= 0), in_stock INTEGER DEFAULT 0, published DATE, FOREIGN KEY (author_id) REFERENCES authors(author_id) ); -- 2) Insert parents first (their rows must exist before books reference them) INSERT INTO authors (author_id, name, country) VALUES (1, 'Ursula K. Le Guin', 'US'), (2, 'Haruki Murakami', 'JP'), (3, 'Chinua Achebe', 'NG'); -- 3) Insert children — defaults & AUTOINCREMENT fill the rest INSERT INTO books (title, author_id, price, published) VALUES ('A Wizard of Earthsea', 1, 9.99, '1968-01-01'); INSERT INTO books (title, author_id, genre, price, in_stock, published) VALUES ('The Left Hand of Darkness', 1, 'Sci-Fi', 12.50, 8, '1969-03-01'), ('Kafka on the Shore', 2, 'Fiction', 14.00, 5, '2002-09-12'), ('Things Fall Apart', 3, 'Fiction', 10.25, 12, '1958-06-17'); -- 4) Copy a filtered set into a new table CREATE TABLE cheap_books (title TEXT, price REAL); INSERT INTO cheap_books (title, price) SELECT title, price FROM books WHERE price < 11; -- 5) Check it worked: each book with its author SELECT b.title, a.name AS author, a.country, b.price FROM books b JOIN authors a ON a.author_id = b.author_id ORDER BY b.price DESC;

Try it

Run the script above in the runner, then write a query to list only the Fiction books along with their author's name, most expensive first.

Show answer
SELECT b.title, a.name AS author, b.price FROM books b JOIN authors a ON a.author_id = b.author_id WHERE b.genre = 'Fiction' ORDER BY b.price DESC;

The final SELECT in the script returns four books; this one filters to the two Fiction titles (Kafka on the Shore, then Things Fall Apart).

Good habits

A few things that trip people up

Single vs double quotes — text values use 'single quotes'. Double quotes are for identifiers (column/table names), not strings.
= vs LIKE — = is exact. For partial matches use LIKE 'SD-WAN%' (% = any characters, _ = one character).
NULL is special — use IS NULL / IS NOT NULL, never = NULL. Here, actual_go_live is NULL for orders not yet delivered.
Integer division — 3/2 can return 1. Multiply by 1.0 (e.g. 100.0 * a / b) to force decimals.
COUNT(*) vs COUNT(col) — COUNT(*) counts rows; COUNT(col) ignores NULLs in that column.
