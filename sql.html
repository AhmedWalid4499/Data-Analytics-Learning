Orientation

What SQL is, and where it sits

SQL (Structured Query Language) reads and writes data in relational tables — the rows-and-columns model from the Foundations page. As an analyst you'll mostly write SELECT queries: pick columns, filter rows, group and aggregate. Master that and you can answer almost any "how much / how many / by what" question without waiting on anyone.

The path

Five levels, basics to advanced

You don't need all of SQL at once. Climb these five levels in order — the lessons below follow the same path, and everything is runnable in the in-browser engine.

Level 1 · Read data — SELECT, FROM, WHERE, ORDER BY, LIMIT. Pull columns, filter rows, sort, and cap the output.
Level 2 · Summarise — aggregates (COUNT, SUM, AVG, MIN, MAX), GROUP BY, column aliases with AS, and DISTINCT. Turn many rows into one number per category.
Level 3 · Filter & shape — HAVING, pattern matching with LIKE, IN/BETWEEN, NULL handling, and CASE for if-then logic.
Level 4 · Combine tables — JOIN (inner and LEFT JOIN) to bring two tables together on a shared key.
Level 5 · Advanced — subqueries, common table expressions (WITH), and window functions (RANK() OVER (...)) for ranking and running totals without collapsing rows.

The core clauses

Build a query, one clause at a time

SQL reads almost like English, and the clauses always run in a set order. Copy any example into the runner further down to see it work.

SELECT … FROM — choose columns

Pick the columns you want from a table. * means "all columns".

SELECT customer, region, revenue FROM orders;

WHERE — filter rows

Keep only the rows that match a condition. Text goes in single quotes.

SELECT customer, revenue FROM orders WHERE region = 'EMEA' AND status = 'Delivered';

ORDER BY & LIMIT — sort and cap

Sort by a column (DESC for high-to-low) and keep just the top few.

SELECT customer, site_code, revenue FROM orders ORDER BY revenue DESC LIMIT 5;

Aggregates + GROUP BY — summarise

Functions like COUNT, SUM, AVG, MIN, MAX collapse many rows into one. GROUP BY does it per category. AS renames a column.

SELECT region, COUNT(*) AS orders, ROUND(SUM(revenue)) AS total_rev, ROUND(AVG(fte_days),1) AS avg_fte FROM orders GROUP BY region ORDER BY total_rev DESC;

HAVING — filter groups

WHERE filters rows before grouping; HAVING filters the groups after.

SELECT dpm, COUNT(*) AS orders FROM orders GROUP BY dpm HAVING COUNT(*) >= 10 ORDER BY orders DESC;

JOIN — combine tables

Match rows from two tables on a shared key. Here we compare each region's actual revenue to its target.

SELECT o.region, ROUND(SUM(o.revenue)) AS actual, t.revenue_target, ROUND(100.0 * SUM(o.revenue) / t.revenue_target) AS pct_of_target FROM orders o JOIN targets t ON o.region = t.region GROUP BY o.region ORDER BY actual DESC;
i
Order of execution: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. That's why you can't use a SELECT alias inside WHERE — the alias doesn't exist yet at that point.
Levels 3 & 5 · go deeper

Shaping logic and advanced queries

Once the core clauses feel natural, these four take you most of the rest of the way. All of them run in the engine below.

CASE — if-then logic in a column

Bucket rows into categories on the fly, then group by the result.

SELECT CASE WHEN revenue >= 30000 THEN 'large' WHEN revenue >= 15000 THEN 'mid' ELSE 'small' END AS band, COUNT(*) AS orders FROM orders GROUP BY band ORDER BY orders DESC;

Subqueries — a query inside a query

Use the result of one query inside another. Here: orders above the overall average revenue.

SELECT customer, site_code, revenue FROM orders WHERE revenue > (SELECT AVG(revenue) FROM orders) ORDER BY revenue DESC;

CTE (WITH) — name a step for readability

A common table expression is a named temporary result you can build on — cleaner than nesting subqueries.

WITH region_rev AS ( SELECT region, SUM(revenue) AS total FROM orders GROUP BY region ) SELECT region, total FROM region_rev ORDER BY total DESC;

Window functions — rank without collapsing rows

Aggregate across rows while keeping every row. RANK() OVER (...) ranks each order by revenue within its region.

SELECT region, customer, revenue, RANK() OVER (PARTITION BY region ORDER BY revenue DESC) AS rev_rank FROM orders ORDER BY region, rev_rank LIMIT 15;

Good habits

A few things that trip people up

Single vs double quotes — text values use 'single quotes'. Double quotes are for identifiers (column/table names), not strings.
= vs LIKE — = is exact. For partial matches use LIKE 'SD-WAN%' (% = any characters, _ = one character).
NULL is special — use IS NULL / IS NOT NULL, never = NULL. Here, actual_go_live is NULL for orders not yet delivered.
Integer division — 3/2 can return 1. Multiply by 1.0 (e.g. 100.0 * a / b) to force decimals.
COUNT(*) vs COUNT(col) — COUNT(*) counts rows; COUNT(col) ignores NULLs in that column.
